## ⚠️ Non-negotiable rules (read before doing anything)

1. **Token Gate** — never use a hardcoded color, spacing, radius, or duration. Every value must come from a CSS custom property (`var(--…)`) defined from the spec. If a token is missing, **add it first** at the correct layer (`ref → sys → comp`); do not inline a hex/rgb/px workaround in a component or screen.
2. **Composition Gate** — before creating a new component, run the **Component Reuse Gate** in `skills/ui-screenshot-to-storybook-product/SKILL.md` (Phase 2). Reuse or extend existing components; do not duplicate just because a clip uses a different name.
3. **Storybook Gate** — every reusable UI block needs a co-located `.stories.tsx` (Default + Variants + States) reviewed in Storybook **before** it is imported into a product screen. Product screens **only compose** documented components; no inline design-system markup in route files.
4. **Reference Fidelity** — match the screenshots' density, restraint, surface model, and information hierarchy. **Do not** add hero sections, decorative gradients, glassmorphism, glow effects, oversized empty cards, or "AI demo" flourishes unless the reference clearly shows them.
5. **Semantic chart colors** — `--color-bullish` (price up) and `--color-bearish` (price down) are independent of `--color-primary` and `--color-error`. Never substitute.

## How to use the skills (Codex-specific)

Codex has no slash commands and no `@file` syntax. To invoke a skill, **explicitly instruct yourself** with a phrase like:

> *"Follow the workflow in `skills/ui-screenshot-to-storybook-product/SKILL.md` to implement `design-reference/screen-0.png`. Run Phase A (visual inventory) first; do not write code until Phase B (component list) is approved."*

For visual comparison or audit:

> *"Follow `skills/ui-visual-parity/SKILL.md` to compare `design-reference/screen-1.png` against the matching screen in `src/pages/`. Run Discovery → Target Resolution → Comparison + Precision, then apply the Component-First Fix Strategy in layered order (token → primitive → variant → composition → page-only). Cite evidence per fix. Never inline a hardcoded value to make a visual match."*

**Reading screenshots:** if your Codex mode is vision-capable, include the file path in the prompt. If not, ask the user to describe the screenshot or paste OCR text — do not invent visual details.

## Workflow — Phase A → E (mandatory order)

| Phase | Gate | Output |
|-------|------|--------|
| **A — Visual inventory** | No code until B approved | Markdown table per screenshot: block / color roles / spacing tier / typography tier / interactive / component mapping |
| **B — Component list** | Confirm list before C | `\| Component \| Responsibility \| Screens \|` plus a Component Reuse Gate table |
| **C — Storybook-first** | Story passes visually before screen use | `src/components/Name.tsx` + `Name.stories.tsx` (Default, AllVariants, AllStates) |
| **D — Screen composition** | Only Phase C imports allowed | `src/pages/*.tsx` or `src/screens/*.tsx` |
| **E — Token audit** | Zero hardcoded values | `grep -rE '#[0-9a-fA-F]{3,6}' src/components src/pages` returns nothing |

Stop after Phase B and ask the user to confirm the component list — unless the user explicitly said *"proceed autonomously"*. Before stopping, update `start-here/SESSION_STATE.md` with the completed phase, key decisions, files changed, verification results, open questions, and recommended next prompt.

## Phase F — UI Visual Parity Check *(建議在 Phase E 後執行)*

Phase E 完成後，建議進行視覺比對，確認實作結果與 `design-reference/` 截圖的對齊程度。

**Codex 執行方式（無斜線指令）：**
```
Read and follow skills/ui-visual-parity/SKILL.md.
Compare the implementation against every screenshot in design-reference/.
Output a discrepancy table: | Block | Expected (reference) | Actual | Difference | Fix |
Apply fixes using only CSS custom properties — no hardcoded values.
```

**💡 GPT-5.5 推薦：** GPT-5.5（vision 版本）在同時分析多張截圖、辨識細微色差（5–10%）與間距偏差（2–4px）方面有顯著優勢。建議在此階段切換至 GPT-5.5 以獲得更精準的比對結果。

**選擇：**
- **A)** 執行 ui-visual-parity 全畫面比對
- **B)** 執行 ui-visual-parity 指定畫面：tell me which screen name
- **C)** 跳過，直接交付

## When in doubt

- **Conflict between screenshot and `CLAUDE.md`** → prefer the token spec + governance.
- **Missing token** → add it at the correct layer (`ref → sys → comp`) before using.
- **Ambiguous component name** (e.g. `DateTab`) → run the Reuse Gate; usually maps to an existing `Tab`/`SegmentedControl`/`ChipGroup`.
- **PRD is placeholder** → continue with the visual/reference workflow but do not invent major product behavior.
- **Real PRD provided** → run the `/prd` workflow described in `product/docs/product-change-workflow.md` and reconcile manifests before Phase A.
