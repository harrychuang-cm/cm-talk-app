### Enforcement rules for this project
- **Token Gate**: never use a hardcoded color, spacing, radius, or duration value. All values must come from CSS custom properties you define from this spec (see also `./design/extracted-design-tokens/design-tokens.json`).
- **Composition Gate**: reuse small components in your app; avoid one-off duplicates of large patterns without a token-backed plan.
- **Storybook Gate**: every reusable UI block must have a co-located `.stories.tsx` and be reviewed in Storybook (same theme/tokens as the app) **before** it is used in product screens. Screens only **compose** Storybook-documented components to match reference screenshots.
- **Storybook MCP Gate**: install `@storybook/addon-mcp`, connect the local `/mcp` endpoint to the agent with `mcp-add --scope project`, and use Storybook MCP docs/story/test tools before deciding component props or marking stories complete.
- **Design System Architect + Foundations**: research tokens in this spec first; in Storybook, add **Foundations** documentation — Color, Typography, Spacing/layout, Shape/radius, Elevation/shadow, and Iconography — as first-class pages beside component stories. Author every foundation guide as a full-screen `.tsx` documentation page plus a minimal CSF `.stories.tsx` route entry. Do not use `.jsx`, `.mdx`, or autodocs for foundations. Each guide must read like a design-system website page: explain design intent, token contract, role mapping, usage timing, accessibility expectations, implementation examples, and do/don't guidance. Use an editorial documentation layout, not a plain card grid: a full-width hero, sticky in-page section rail, dense token tables, visual specimens, implementation notes, and paired do/don't examples. Keep cards only for individual specimens or examples. Foundation `.stories.tsx` files are Storybook navigation shells only: import `Meta`/`StoryObj`, set `title: 'Foundations/<Name>'`, use `parameters: { layout: 'fullscreen' }`, render the guide page, and do not set `tags: ['autodocs']`. If a project enables autodocs globally, explicitly opt these stories out with `tags: ['!autodocs']`. Use these fixed documentation page templates:
- **Color**: semantic color model, surface ladder, role swatches with resolved values, interaction/state colors, finance semantics, accessibility contrast notes, implementation snippets, do/don't.
- **Typography**: type scale specimens, role-to-UI mapping, hierarchy rules for dense screens, truncation/overflow rules, implementation snippets, do/don't.

### Phase A — visual analysis rules

**Color token mapping from screenshots:**
- Darkest canvas area → `--color-background`
- Card / sheet surface (one visible step lighter) → `--color-surface-container`
- Brightest interactive element (CTA button, active tab indicator) → `--color-primary`
- Primary body text on dark surface → `--color-on-surface`
- Secondary / metadata text (dimmer, ~60% opacity) → `--color-on-surface-variant`
- Price-up highlight (green) → `--color-bullish` — **never substitute `primary`**
- Price-down highlight (red) → `--color-bearish` — **never substitute `error`**

**Spacing tier from row height:**
- ~48 px row height (tab bar, action strip, compact list) → **compact**
- ~64 px row height (asset list, news row, settings row) → **standard**
- 120 px+ block height (balance hero card, chart area) → **hero**

**Typography tier from visual weight/size:**
- Largest boldest number (portfolio balance, main price) → `displaySmall` / `headlineLarge`
- Section title ("Holdings", "Markets") → `titleMedium` or uppercase `labelLarge`
- Row body text (asset name, description line) → `bodyMedium` / `bodySmall`
- Timestamps, secondary metadata, % change labels → `labelSmall`

**Required Phase A output format (one row per visual block):**
`| Block | Color roles | Spacing tier | Typography tier | Interactive? | Component mapping |`

### Reference Fidelity Protocol — avoid AI-demo drift

Before Phase B, inspect every reference screenshot as a product artifact, not as loose inspiration. The implementation must preserve the reference's density, restraint, and information hierarchy.

For each screenshot, explicitly capture:
- **Layout anatomy**: fixed header/footer/nav, scrollable regions, overlays, sheets, cards, list rows, and empty space.
- **Alignment system**: left/right alignment, numeric alignment, icon-text baselines, column widths, and repeated gutters.
- **Surface model**: which blocks are truly cards/sheets, which are flat page sections, and where borders or elevation actually appear.
- **Content realism**: visible row counts, label lengths, number/date formats, avatar/logo behavior, truncation, and secondary metadata.
- **Interaction evidence**: active tab/nav state, selected chips, disabled affordances, pressed states, badges, menus, and focus targets.
- **Component boundaries**: reusable component vs. one-off layout wrapper; do not invent new wrappers just to make the UI look polished.

Anti-demo rules:
- Do not add hero sections, marketing copy, decorative gradients, glassmorphism, glow blobs, oversized empty cards, or illustration panels unless the reference clearly contains that pattern.
- Do not wrap every section in a card. Card/sheet treatment is allowed only where the reference shows containment through background, border, elevation, or spacing.
- Do not "beautify" by increasing whitespace, rounding, shadows, blur, saturation, or icon density beyond the reference.
- Do not use lorem ipsum, generic SaaS slogans, or symmetrical placeholder data. Use realistic domain data with the same visual length and density as the screenshots.
- Do not replace a dense operational/product screen with a landing page or dashboard template. The first screen should be the actual referenced product UI.
- Visual polish must come from matching alignment, typography, spacing rhythm, token usage, and real states — not from generic AI decoration.

Background images / photos rule (raster content):
- When the reference shows a **photo, hero banner, profile picture, illustration, or any raster artwork**, do NOT recreate it with CSS gradients, SVG, emoji, or hand-coded shapes. Do NOT use third-party hotlinked placeholders (`lorem-picsum`, `placeholder.com`, `unsplash random`).
- **Default**: replace with a token-driven `background-color` placeholder (closest existing surface role: `--color-surface-container` / `--color-placeholder` / equivalent — never a literal hex). Preserve width, height, aspect-ratio, border-radius, and spacing. Add a discoverable marker: `data-placeholder="image"` + `role="img"` + meaningful `aria-label` so the user can swap real images later.
- **Auto-generation** is allowed only when the project declares image-generation tooling (npm script / MCP server / documented API endpoint) AND the required API key is in `.env`. Otherwise stop at the placeholder.
- See `skills/ui-visual-parity/SKILL.md` → **Background Images and Photos** for the full decision tree and summary-table format.

Mobile rules (apply always at this platform setting):

Mobile-first viewport contract (mobile app target — NO horizontal page scroll):
- This pack scaffolds a **mobile-shaped product**. Every screen MUST fit the viewport width — `document.documentElement.scrollWidth === document.documentElement.clientWidth` at the smallest target width (e.g. 360px).
- App shell root: `width: 100vw; overflow-x: hidden`. Page body scrolls **vertically only**; no element may push `<body>` to scroll horizontally.
- Wide lists (horizontal carousels, chip rows, tab strips, horizontal tables): the **list container** owns its own `overflow-x: auto` inside a `max-width: 100%` parent. Horizontal overflow stays **inside** the bounded list region — it must never bubble up to the page or body.
- Overflow-prone content (long strings, unbroken numbers, code blocks, images, tables): enforce `max-width: 100%`, `word-break: break-word`, `overflow-wrap: anywhere`, `object-fit: contain`, or `table-layout: fixed` so it cannot blow out the layout.
- Pre-ship check: open DevTools at 360px (and the largest target width), confirm no horizontal scrollbar on `<html>` / `<body>`, and that any visible horizontal scrolling lives inside a clearly bounded list/carousel.

Top App Bar center anchoring (middle slot must be screen-centered):
- The middle slot of the top app bar (title text or center action) MUST be anchored to the **visual center of the screen** — not to the center of the leftover flex space between the left and right slots.
- Asymmetric side slots (e.g. left = back button + breadcrumb, right = empty) MUST NOT cause the title to drift. Implement with one of:
  - **Absolute center**: position: absolute; left: 50%; transform: translateX(-50%) on the middle slot, with left/right slots positioned absolutely on their edges; OR
  - **3-cell grid**: grid-template-columns: 1fr auto 1fr with symmetric horizontal padding on the side cells; OR
  - **Mirrored side widths**: render both left and right slots inside equal-width containers (the empty side still occupies its slot), so the middle naturally falls at 50%.
- Long titles in the centered slot use text-overflow: ellipsis + max-width constrained by the available middle column — they MUST NOT push the slot off-center or wrap into multiple lines.
- Exception: when the reference explicitly shows a left-aligned title style (e.g. iOS large title pattern, certain Material 3 variants), follow the reference. Do not retro-fit centered titles onto a clearly left-aligned reference.

BottomNav anchoring (fixed to viewport bottom):
- The BottomNav MUST be anchored to the bottom of the viewport for the entire app session — it does NOT scroll away with page content. Implement with `position: fixed; left: 0; right: 0; bottom: 0` on the nav, OR by composing the app shell as a flex column where BottomNav is the last child of a `height: 100vh` (or `100dvh`) shell with page content in a `flex: 1; overflow-y: auto` middle region.
- Every screen's main scroll container MUST reserve bottom space equal to the BottomNav height (e.g. `padding-bottom: var(--bottom-nav-h)` where the nav height is a token), so the last row of content is not hidden behind the nav.
- Honor iOS / Android safe areas: add `padding-bottom: env(safe-area-inset-bottom)` (or `max(env(safe-area-inset-bottom), 0px)`) to the BottomNav inner container so the touch targets are not under the home indicator.
- BottomNav surface is opaque (token-driven background) — do NOT rely on a transparent or blurred bar that lets page content show through unless the reference shows that exact treatment.
- The BottomNav stays visible across modal sheets unless the reference shows a full-screen modal that covers the bar; full-screen modals MUST hide the nav via render condition, not by overlapping.
- Exception: a screen explicitly designed without bottom navigation (e.g. onboarding splash, full-screen camera, immersive media viewer) may omit the BottomNav — but the surrounding app shell still anchors it for every other screen.

Card / Tab border discipline (avoid AI-slop look):
- **Cards:** default to **borderless / outline-less**. Communicate containment via background contrast (`--surface-2`, `--surfaceContainer`), spacing rhythm, or a single elevation level — NOT via 1–2px outlines on every card. A border on a card is allowed only when (a) the reference clearly shows one, (b) the card sits on a surface of identical color and there is no other way to show containment, or (c) the role demands an explicit boundary (selectable / draggable card with active state).
- **Tabs:** default to **borderless** tab strips. Show active state via underline on the active tab only, weight change, or solid pill fill — not by drawing borders around every tab. The container holding the tab strip should NOT have an outer border; if separation from page content is needed, use a single hairline divider on one edge only.
- **Inputs / buttons / chips** keep their borders as defined in the component spec — the discipline above applies specifically to surface/grouping containers.
- Symptom of AI-slop you must avoid: every card AND every tab AND every section having a 1px outline at once. If three or more nested containers all have visible borders, remove borders from the inner two and rely on background or spacing instead.

List-item container restraint (don't wrap every row in a card):
- **Lists default to flat rows.** Do NOT wrap every list item in its own card / outlined container by reflex. The default treatment is rows of content separated by spacing or a single hairline divider, anchored by typographic hierarchy.
- Wrap a row in a container ONLY when one of these is true: (a) the reference clearly shows row-as-card; (b) the row is **a single tappable target as a whole** AND elevation/border meaningfully communicates that affordance beyond color/hover; (c) the row contains genuinely composite content (image + title + meta + multi-action) that cannot be grouped by spacing alone.
- If you find yourself rendering 3+ adjacent rows that all look like identical bordered/elevated cards with the same padding while the reference shows them as flat rows with dividers, **remove the card wrapper** — keep only the inner content + a single hairline divider.
- Avoid double-containment: a list inside a card should never have each row also inside a card; pick one level only.

- `--color-negative` (bearish): price decreases, negative P&L, downward trend indicators. ✗ Do NOT use `--color-error` for bearish values.

**Typography Guide — per-scale application scenarios (include in the Usage section):**

Map each M3 scale role to concrete UI contexts:
- `displayLarge` (57px): Page hero title, brand moment, onboarding splash. Rarely used — once per screen maximum.
- `displayMedium` / `displaySmall`: Large numeric displays (portfolio value, stock price in hero), major feature headers.
- `headlineLarge` / `headlineMedium`: Section titles, screen titles in app bars, modal headings.
- `headlineSmall`: Card section headers, group labels above lists.
- `titleLarge` / `titleMedium`: Card titles, list item primary text, dialog titles. `titleMedium` is the most common "heading within a component" size.
- `titleSmall`: Sub-section labels, tab labels, chip labels in larger chips.
- `bodyLarge` / `bodyMedium`: Primary content text — descriptions, paragraphs, row primary information. `bodyMedium` is the default reading text size.
- `bodySmall`: Secondary information, help text, timestamps in dense lists.
- `labelLarge`: Button text, prominent form field labels.
- `labelMedium`: Form labels, input placeholder text, badge text.
- `labelSmall`: Metadata, chip text, caption below images, footnotes. Use when space is very tight and the text is supplementary.

**Spacing Guide — interval logic (include in the Usage section):**

Explain the spacing scale as a purposeful system, not arbitrary numbers. Reference the base unit from Step 0-A-2 (4px or 8px):

- **xs** (= 1× base): Intra-component element gaps — the space between an icon and its label inside a button, between two inline badges.
- **sm** (= 2× base): Component internal padding — horizontal/vertical padding inside small components (chips, tags, small buttons).
- **md** (= 4× base): Related component group spacing — vertical gap between a form label and its input, horizontal gap between sibling buttons in a toolbar.
- **lg** (= 6× base): Section spacing — vertical rhythm between distinct content groups within a screen section (e.g., between a heading and its list).
- **xl** (= 8× base): Page-level section separation — the gap between major screen sections (e.g., hero → feature list → CTA block).
- **xxl** (= 12× base): Extra-generous section separation for hero layouts or promotional screens.

Include a multiplication table in the guide showing all scale steps and their px values. Label which density classification (spacious/balanced/compact) each value is associated with.

   - **Authoritative:** the **Component corner radius** table, the theme **SYS ladder** (`--radius-sm` … `--radius-xl`, especially `--radius-md` for **buttons / primary controls**), `--radius-cmp-*` aliases, and each **Border-radius (implement this)** line — **must agree** per role.
   - **Reference only:** MD3 **Shape** code block / **Shape note** describe extraction context; they do **not** override the table. If the reference shows rounded UI, set Explorer **圓角** presets so the table and theme show non-`none` values — do **not** hardcode `0px` on primitives while vars stay rounded.
   - **Card vs controls:** card-like surfaces → `--radius-lg` / `--radius-cmp-card`; standard controls / buttons → `--radius-md` / `--radius-cmp-button`. Pills → `--radius-full` when the chip role is `pill`.
5. **Type hierarchy**:  ALWAYS apply `font-variant-numeric: tabular-nums` on any numeric value.
6. **Interaction cues**: 
7. **Both themes**: Implement full dark AND light theme using the token sets above. A theme toggle must switch ALL colors simultaneously — no hardcoded color values.
8. **Surface depth**: Use the 4-level surface hierarchy — background → surface → surfaceContainer → surfaceContainerHigh. Adjacent steps must read as different surfaces (not 1–2% tweaks only). **Shadows vs flat:** Follow the reference: surface steps + hairline borders for dense shells; elevation shadows for true floats (FAB, dialogs), not every row.
9. **Icons — via shared Icon component**: Build a reusable `Icon` wrapper in the shared component layer. Source icons from `lucide-react` by default, set `strokeWidth={2}` (1.5 for decorative/large icons), and route decorative vs informative accessibility through that wrapper. Do NOT scatter inline SVG across product screens.
10. **Charts — use charting libraries, not hand-drawn SVG** (only if charts are visible in the references):
    - **Candlestick + Volume**: `lightweight-charts` with bullish/bearish tokens
    - **Sparklines**: `recharts` `<AreaChart>` with gradient fill per sparkline spec above
    - **Portfolio sparkline**: `recharts` `<AreaChart>` with gradient fill, no axes
    - NEVER use raw `<svg>` / `<polyline>` for charts. All charts must be responsive (`ResponsiveContainer` or `width="100%"`).
11. **Data formatting**: Use `Intl.NumberFormat` for numeric domain values. For non-finance products, format dates, counts, currencies, ratings, and status labels according to the visible reference context; do not invent price/P&L semantics.
12. **No hollow UI**: Every screen must have enough content density to feel complete — no large empty gray areas. Use section headers, cards, lists, badges, metadata, and reference-specific media/content to fill visual space with meaning.
13. **Asset color system**: For avatars, logos, thumbnails, or category icons visible in the reference, use branded or token-derived fallback colors. Do not use generic gray initials unless the reference itself does.
14. **Image error handling**: ALWAYS handle `onError` on `<img>` tags with a gradient placeholder. Never show broken image icons.
15. **Mobile-first viewport contract — NO horizontal page scroll** (mobile app target):
    - App shell root: `width: 100vw; overflow-x: hidden`. Every screen MUST fit the viewport width.
    - Page body scrolls **vertically only**. NO element may cause `<body>` / `<html>` to scroll horizontally on the smallest target width (e.g. 360px).
    - Wide lists (horizontal carousels, chip rows, tab strips, horizontal tables) MUST own their own `overflow-x: auto` **inside** a `max-width: 100%` container — the horizontal scroll stays inside the bounded list region and never bubbles up to the page.
    - Overflow-prone content (long strings, unbroken numbers, code blocks, images, tables): enforce `max-width: 100%`, `word-break: break-word`, `overflow-wrap: anywhere`, `object-fit: contain`, or `table-layout: fixed` so it cannot push the layout wider than the viewport.
    - Pre-ship check: at 360px width, `document.documentElement.scrollWidth === document.documentElement.clientWidth` AND no horizontal scrollbar on `<html>` / `<body>`.
16. **Card / Tab border discipline — avoid AI-slop look**:
    - **Cards default to borderless.** Communicate containment via background contrast (`--surface-2` / `--surfaceContainer`), elevation, or spacing rhythm — not via a 1–2px outline on every card. A border on a card is allowed only when the reference clearly shows one, the card sits on a same-color surface with no other separation cue, or the card has a selectable / draggable role that needs an explicit boundary.
    - **Tabs default to borderless.** Express the active tab via underline on the active tab only, weight / color change, or a solid pill fill — never by outlining every tab. The strip container itself MUST NOT have an outer border; if separation from page content is needed, use a single hairline divider on one edge only.
    - Inputs, buttons, chips keep the borders defined in their component spec — this rule targets surface / grouping containers.
    - Anti-pattern: every card AND every tab AND every section all carry a visible 1px outline. If three or more nested containers each have a border, remove borders from the inner two and use background or spacing instead.
17. **List-item container restraint — don't wrap every row in a card**:
    - **Lists default to flat rows** with typographic hierarchy + spacing + at most one hairline divider. Do NOT reflexively wrap every list item in its own card / outlined container.
    - Wrap a row in a container ONLY when (a) the reference shows row-as-card, (b) the row is a single tappable target as a whole AND elevation/border genuinely communicates that affordance, or (c) the row carries composite content (image + title + meta + multi-action) that cannot be grouped by spacing alone.
    - If 3+ adjacent rows all look like identical bordered/elevated cards while the reference renders them as flat rows with dividers, **delete the card wrapper** and keep only the inner content + one `border-bottom` divider.
    - Never double-contain: a list already inside a card MUST NOT have each row also inside a card; pick one level only.
18. **Top App Bar center anchoring — middle slot is screen-centered**:
    - The center slot of the top app bar (title or center action) MUST be anchored to the **visual center of the screen** — not to the center of the leftover flex space between left and right slots. Asymmetric side slots (e.g. left = back + breadcrumb, right = empty) MUST NOT cause the title to drift.
    - Implement using one of: **(a)** absolute center: `position: absolute; left: 50%; transform: translateX(-50%)` on the middle slot with side slots positioned absolutely on their edges, OR **(b)** 3-cell grid: `grid-template-columns: 1fr auto 1fr` with symmetric horizontal padding, OR **(c)** mirrored side containers of equal width (empty side still occupies its slot).
    - Long titles in the center use `text-overflow: ellipsis` + a max-width constrained by the middle column — they MUST NOT push the slot off-center or wrap.
    - Exception: when the reference clearly shows a **left-aligned** title pattern (iOS large title, certain Material 3 variants), follow the reference. Do not retro-fit centered titles onto a left-aligned reference.
19. **BottomNav anchoring — fixed to viewport bottom**:
    - BottomNav MUST be anchored to the bottom of the viewport for every screen that uses it — it does NOT scroll away with page content. Implement with `position: fixed; left: 0; right: 0; bottom: 0` on the nav, OR compose the app shell as a flex column on a `height: 100dvh` shell where the page content region is `flex: 1; overflow-y: auto` and BottomNav is the last child.
    - Every page's main scroll container MUST reserve bottom space equal to the BottomNav height (e.g. `padding-bottom: var(--bottom-nav-h)`) so the last row of content is never hidden under the nav.
    - Honor mobile safe areas: add `padding-bottom: env(safe-area-inset-bottom)` (or `max(env(safe-area-inset-bottom), 0px)`) to the BottomNav inner container so touch targets sit above the iOS home indicator / Android gesture bar.
    - BottomNav surface is opaque (token-driven background); do not use a transparent or blurred bar that lets page content show through unless the reference clearly shows that treatment.
    - Full-screen modals MUST hide the BottomNav via render condition (do not let the modal overlap the bar). The bar stays visible across regular sheets / dialogs unless the reference shows otherwise.
    - Exception: screens explicitly designed without bottom navigation (onboarding splash, full-screen camera, immersive media viewer) may omit BottomNav, but the surrounding app shell still anchors it on every other screen.

### Step 0 — Token layer: ref → sys → comp  (MANDATORY — Google MD3 inheritance)

Design tokens follow **Google Material Design 3** three-layer inheritance. **Never skip a layer** — components may only consume `--comp-*` variables:

```
ref  (--ref-*)   raw palette steps + scale primitives        ← never changes with theme
      ↓ referenced by
sys  (--sys-*)   semantic roles per [data-theme]             ← only color roles change per theme
      ↓ referenced by
comp (--comp-*)  per-component slots → ref --sys-* only      ← :root, not theme-scoped
      ↓ consumed by
Components / TSX   read only --comp-* — never --sys-*, --ref-*, or raw hex
```

**File structure** — split into three files, barrel-imported in order:
```
src/styles/
  tokens-ref.css   ← :root  raw primitives (extracted palette + scale)
  tokens-sys.css   ← [data-theme]  semantic roles
  tokens-comp.css  ← :root  per-component slots (→ sys only)
  tokens.css       ← barrel: @import all three in order
```

```ts
// src/main.tsx  and  .storybook/preview.tsx — import the barrel

  --comp-dropdown-shape:       var(--sys-shape-sm);
  --comp-dropdown-height:      44px;
}
```

#### Barrel — `tokens.css`
```css
/* tokens.css */
@import './tokens-ref.css';
@import './tokens-sys.css';
@import './tokens-comp.css';
```

Set `data-theme="dark"` (or `"light"`) on `html` / `#root` to activate the correct system layer.

> **Token enforcement:**
> - **Add a new visual property to a component?** Add a `--comp-<component>-<slot>` pointing to a `--sys-*` in `tokens-comp.css`, then use `var(--comp-…)` in TSX.
> - **Change a theme color?** Edit the color roles in `tokens-sys.css` only.
> - **Change a brand palette hex?** Edit `tokens-ref.css` only — propagates automatically through sys → comp.
> - **Change a component's shape, height, or padding?** Edit `tokens-comp.css` only.
> - **Never** write raw hex, `--ref-*`, or `--sys-*` directly inside component TSX or CSS.

### Step 1 — Storybook setup (MANDATORY — before product screens)
1. Initialize Storybook v10 in the project root: `npx storybook@10 init` (framework auto-detected; no `--type` flag needed). Install addons: `npm install -D @storybook/addon-themes @storybook/addon-a11y` and `npx storybook add @storybook/addon-mcp` — the Docs addon is **built-in** in v10, no separate `@storybook/addon-docs` install needed.
2. In `.storybook/preview.tsx`, import **the same** `tokens.css` / global styles as `main.tsx`. Configure theme switching with `withThemeByDataAttribute` (or equivalent) so **`data-theme` values match the app exactly**.
3. Start Storybook in one terminal and connect the MCP endpoint from another terminal: `npx storybook dev`, then `npx mcp-add --type http --url "http://localhost:6006/mcp" --scope project` (change `6006` if Storybook uses a different port). Choose a unique server name such as `<app-name>-storybook-mcp`.
4. **MCP docs rule:** before using any component prop or variant, query `list-all-documentation` and `get-documentation` / `get-documentation-for-story`. Before writing or updating stories, query `get-storybook-story-instructions`. After story changes, run `run-story-tests` where available.
5. **Component autodocs rule:** reusable component story files under `src/components/` must set `tags: ['autodocs']` on the default export meta — the Docs addon generates prop tables and usage examples for component APIs. Foundation guide stories are the exception: they are full-page documentation routes and must not use autodocs.
6. **Storybook Gate:** no new reusable visual block ships in the app until it has a **Story** — Storybook is the catalog the team uses to review components in isolation.
